## Section 1: Conditional simplify step

**Status:** [x] complete
**Model:** haiku
**User stories covered:** 1, 2, 3

### What to build

Add **Step 6a** to `skills/build/SKILL.md`, inserted between the existing Step 6 (completion announcement) and the new Step 6b. Step 6a determines whether the branch has any non-markdown code changes; if so, it invokes `/simplify` via the Skill tool scoped to those files.

### Acceptance criteria

- [x] When all sections are complete and the branch contains at least one non-`.md` changed file, `/build` invokes `/simplify` via the Skill tool with the changed non-md file list passed as `args`.
- [x] When all sections are complete and the branch contains only `.md` file changes, `/build` skips Step 6a silently (no message, no invocation).
- [x] The `/simplify` invocation receives only the non-markdown changed files, not the entire codebase.

### Notes for executor

- Determine changed files with `git diff --name-only $(git merge-base HEAD main) HEAD`; fall back to `master` if `main` is not a valid ref.
- Filter the file list to exclude any path ending in `.md`.
- If the filtered list is empty, skip. If non-empty, invoke Skill with `skill: "simplify"` and `args: "Scope your review to these branch-modified files only: <file list>"`.
- Step 6a must appear in the skill file AFTER the Step 6 completion announcement and BEFORE Step 6b.

## Section 2: Future considerations generation and handoff

**Status:** [ ] not started
**Model:** haiku
**User stories covered:** 4, 5

### What to build

Add **Step 6b** to `skills/build/SKILL.md`, inserted after Step 6a (or after Step 6 if Step 6a was skipped). Step 6b has the controller generate a `## Future Considerations` section, append it to the PRD, commit the PRD update, and print the specified handoff message.

### Acceptance criteria

- [ ] After all sections are complete (and after the optional Step 6a), `/build` generates a `## Future Considerations` section and appends it to the PRD file.
- [ ] The PRD update is committed with the message `docs: add future considerations to PRD`.
- [ ] The handoff message printed is exactly: `Future considerations written to <PRD path>. Run /brainstorm and review the **Future Considerations** section to evaluate next steps.`

### Notes for executor

- The PRD path is already known to the controller from the existing Step 6 announcement — reuse it.
- The `## Future Considerations` section should contain bulleted suggestions generated by the controller based on its knowledge of the completed build: follow-up ideas, edge cases worth revisiting, improvements identified but out of scope. Aim for 3–7 items.
- If the controller can think of no meaningful suggestions, still append the section with a note like `- No additional items identified at this time.` so downstream tooling can always find the section.
- Commit only the PRD file — no other files.
- The handoff message is informational only; do not invoke `/brainstorm` automatically.
